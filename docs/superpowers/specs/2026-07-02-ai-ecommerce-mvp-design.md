# AI 电商系统商用 MVP 改造设计

## 元信息

- 日期：2026-07-02
- 项目：`ecom-flow`
- 状态：已和用户确认，等待实施计划
- 改造路线：整理现有项目，做成可部署的商用 MVP
- 第一版定位：单商家数码电子商城 + AI 运营后台 + 电商工作流自动化

## 1. 背景与目标

用户希望把现有 `ecom-flow` 项目改造成一个适合练习 vibe coding 的 AI 电商系统。系统需要按商用开发步骤推进，而不是只做一个本地演示页面。第一期目标是做出一套能本地运行、能部署到 Linux 服务器、主业务链路完整、代码结构适合学习的商用 MVP。

现有项目已经具备 Next.js、Prisma、NextAuth、商品、订单、工作流、后台页面等雏形，但处于半成品状态：文档和界面文案存在乱码，数据库配置不统一，部分代码结构和功能边界需要整理。本次改造选择保留现有基础，在可控范围内修复并补齐主链路。

第一期成功标准：

- 新用户可以注册、登录、退出。
- 用户可以浏览数码商品、加入购物车、提交订单、完成模拟支付、查看自己的订单。
- 运营可以在后台管理商品、查看订单、处理发货、查看工作流待办。
- 管理员可以管理用户角色、审核商品上架、查看关键工作流日志。
- AI 商品运营助手可以生成商品标题、卖点、详情文案和 SEO 关键词。
- 未配置 AI Key 时系统自动使用模拟 AI 结果，保证本地开发和部署演示不中断。
- 项目可以通过 Docker Compose 部署到普通 Linux 云服务器。

## 2. 范围决策

### 2.1 第一版包含

- 单商家商城，不做多商户。
- 数码电子产品品类。
- 前台商城和独立运营后台。
- 注册登录和三类角色权限：`customer`、`operator`、`admin`。
- 商品、分类、购物车、订单、支付记录、退款、工作流日志。
- 模拟支付，但预留真实支付服务接口。
- 订单流程、退款流程、商品上架流程。
- AI 商品运营助手，支持真实 AI API 和模拟兜底。
- Dockerfile、Docker Compose、`.env.example`、README 部署说明。
- GitHub Actions 至少执行 lint 和 build。

### 2.2 第一版不包含

- 真实支付接入。
- 多商户、多店铺、平台结算和分账。
- 优惠券、积分、评价、收藏。
- 真实物流 API。
- AI 客服机器人。
- 运营数据分析大屏。
- 移动 App。
- 拖拽式工作流编辑器。

这些能力放到第二期或第三期处理，避免第一期复杂度失控。

## 3. 架构设计

第一期采用 Next.js 全栈单体应用。前台页面、后台页面、API、认证、业务逻辑都在同一个项目内。部署时应用作为一个容器运行，数据库和 Redis 由 Docker Compose 管理。

系统分为五层：

1. 商城前台：普通用户浏览商品、购物车、结算、支付、查看订单。
2. 运营后台：运营和管理员管理商品、订单、库存、工作流、AI 文案。
3. API 层：处理前端请求，负责参数校验、权限校验、调用领域服务。
4. 领域服务层：封装业务逻辑，例如 `productService`、`orderService`、`paymentService`、`workflowService`、`aiProductService`。
5. 基础设施层：Prisma、PostgreSQL、Redis 预留、Docker、环境变量、GitHub Actions。

第一期不拆微服务。这样可以让新手更容易理解完整业务闭环，也便于部署、调试和测试。

## 4. 页面与交互设计

### 4.1 商城前台

- `/products`：商品列表，支持分类筛选和商品状态过滤，只展示已上架商品。
- `/products/[id]`：商品详情，展示图片、价格、库存、卖点、详情。
- `/cart`：购物车，支持修改数量、删除商品、进入结算。
- `/checkout`：结算页，填写收货信息，创建订单。
- `/orders`：我的订单，展示订单状态和操作入口。

### 4.2 运营后台

- `/admin`：后台首页，展示订单概览、库存提醒、待办流程、AI 快捷入口。
- `/admin/products`：商品管理，支持新增、编辑、上下架、提交审核。
- `/admin/orders`：订单管理，支持查看订单、推进发货。
- `/admin/workflows`：工作流待办，展示订单、退款、商品审核相关流程。
- `/admin/ai/product-copy`：AI 商品文案工具。

### 4.3 管理员能力

- `/admin/users`：用户列表和角色管理。
- 商品审核：通过或驳回商品上架申请。
- 工作流日志：查看关键业务对象的状态流转记录。
- 系统配置与健康检查：展示数据库、AI 配置、关键环境变量是否可用。

交互原则：前台像正常商城，后台像真实运营工具。后台优先使用表格、筛选、状态标签、操作按钮和详情抽屉，避免做成营销展示页。

## 5. 数据模型设计

第一期围绕以下核心对象建模：

- `User`：用户账号、密码哈希、角色、创建时间。
- `Category`：商品分类，第一期用于数码电子品类。
- `Product`：商品名称、描述、价格、库存、图片、分类、状态、AI 文案字段。
- `CartItem`：用户购物车商品和数量。
- `Order`：订单编号、用户、总金额、收货信息、支付状态、履约状态。
- `OrderItem`：订单商品快照、数量、下单价格。
- `Payment`：支付记录，第一期用于模拟支付，保留外部支付流水字段。
- `Refund`：退款申请、退款金额、原因、审核状态。
- `Workflow`：流程模板。
- `WorkflowInstance`：某个业务对象的一次流程实例。
- `WorkflowLog`：流程流转日志。

订单明细必须保存商品快照，避免商品改价或改名后影响历史订单。

## 6. 工作流设计

工作流第一期采用代码内置流程模板，不做拖拽编辑器。

订单流程：

```text
待支付 -> 已支付 -> 待发货 -> 已发货 -> 已完成
```

退款流程：

```text
已申请 -> 审核中 -> 已退款
已申请 -> 审核中 -> 已驳回
```

商品上架流程：

```text
草稿 -> 待审核 -> 已上架
草稿 -> 待审核 -> 已驳回
```

每次状态推进都要校验当前状态和允许动作，禁止乱跳状态。例如未支付订单不能发货，已完成订单不能重复支付。每次推进都写入 `WorkflowLog`，记录操作者、动作、前后节点、备注和时间。

## 7. AI 商品运营设计

第一期 AI 只做商品运营助手。运营输入商品基础信息后，系统生成：

- 商品标题。
- 3 到 5 个商品卖点。
- 商品详情描述。
- SEO 关键词。
- 一句话推荐理由。

AI 模块分层：

- `aiProvider`：真实 AI API 对接层。
- `mockAiProvider`：无 Key 或调用失败时的模拟结果。
- `aiProductService`：商品文案业务逻辑。
- AI API route：供后台页面调用。

商用安全原则：AI 只生成建议，不直接发布商品。运营需要点击“应用到商品”后，结果才会写回表单或商品记录。

## 8. 支付设计

第一期使用模拟支付，但按真实支付结构预留接口。

支付流程：

1. 用户提交订单，订单状态为待支付。
2. 用户点击模拟支付。
3. 系统创建或更新 `Payment` 记录。
4. 支付成功后订单进入已支付状态，并触发订单工作流推进。
5. 重复支付、已取消订单支付、库存不足等情况必须返回明确错误。

后续接入微信支付、支付宝或 Stripe 时，替换支付 provider 即可，不需要推翻订单模型。

## 9. 权限设计

角色说明：

- `customer`：访问商城前台，管理自己的购物车、订单和退款。
- `operator`：访问运营后台，管理商品、订单、库存、工作流待办。
- `admin`：拥有全部后台权限，可管理用户角色和审核关键流程。

权限必须同时在页面入口和 API 层校验。页面隐藏按钮不能替代服务端权限校验。

## 10. 错误处理设计

必须覆盖以下错误场景：

- 表单输入错误：价格为空、库存为负、邮箱格式不正确。
- 权限错误：普通用户访问后台，运营修改管理员权限。
- 库存错误：库存不足不能下单，支付前再次校验库存。
- 订单状态错误：未支付订单不能发货，已完成订单不能重复支付。
- 支付错误：模拟支付失败、重复支付、支付记录不一致。
- AI 错误：AI Key 未配置、真实 API 调用失败、返回格式不符合预期。
- 工作流错误：无效流转、流程实例不存在、流程已结束。

API 返回统一错误结构，前端用清晰文案提示用户。

## 11. 测试设计

测试按风险优先级安排：

- 单元测试：订单创建、库存扣减、支付状态变化、工作流流转、AI 兜底。
- 集成测试：创建订单、模拟支付、提交退款、商品审核、生成 AI 文案。
- 页面冒烟测试：商品列表、购物车、后台商品列表、订单管理、AI 文案页面可以打开并触发关键操作。
- CI 检查：每次提交至少跑 `npm run lint` 和 `npm run build`。

实施阶段遵循测试先行：新增核心业务逻辑前先写失败测试，再实现功能。

## 12. 部署设计

部署路线：

```text
本地开发 -> GitHub 管理代码 -> GitHub Actions 检查 -> Docker 构建 -> Linux 服务器 -> Docker Compose 启动
```

服务器组件：

- Next.js 应用容器。
- PostgreSQL。
- Redis 预留。
- Nginx 反向代理。
- HTTPS 证书。
- `.env.production` 环境变量。

项目需要提供：

- `Dockerfile`。
- `docker-compose.yml` 本地开发配置。
- `docker-compose.prod.yml` 或生产部署说明。
- `.env.example`。
- README 部署教程。
- GitHub Actions workflow。

## 14. 验收标准

- `customer` 可以完成注册、登录、浏览商品、加购物车、下单、模拟支付、查看订单。
- `operator` 可以进入后台，管理商品，处理订单发货，查看工作流待办。
- `admin` 可以管理用户角色，审核商品上架，查看工作流日志。
- 订单、退款、商品上架都能记录流程日志。
- AI 商品文案在有 Key 时调用真实 AI，无 Key 时返回模拟结果。
- 支付模块使用模拟支付，但代码结构保留真实支付扩展点。
- `npm run lint` 通过。
- `npm run build` 通过。
- Docker Compose 能在本地和 Linux 服务器启动应用、PostgreSQL 和 Redis。
- README 包含从 GitHub 到服务器部署的步骤。

## 15. 后续阶段预留

第二期建议加入 AI 客服助手，基于商品、订单、退款数据回答用户问题。

第三期建议加入 AI 运营分析，分析订单、库存、热销商品、退款异常，并给运营建议。

第四期可以考虑优惠券、评价、真实物流、真实支付、多商户和可视化工作流编辑器。
