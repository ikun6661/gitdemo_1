# EcomFlow — 需求分析书

## 一、项目背景

在向潜在客户展示电商解决方案时，需要一套直观的系统来演示订单从创建到完成的完整自动化流程，以及内部运营的审批流转。当前缺乏一个既能"演示好看"又能"实际使用"的工作流平台。

**核心诉求**：做一个通用工作流引擎，能演示给客户看，也能真正上线运营。

---

## 二、目标用户

| 角色 | 描述 | 核心场景 |
|------|------|----------|
| 演示者（运营/销售） | 向客户展示系统能力 | 打开演示模式，分步展示订单全链路和审批流程 |
| 运营人员 | 日常处理订单和审批 | 在流程看板中查看和处理待办事项 |
| 管理员 | 管理系统基础数据 | 管理商品、用户、审核上架申请 |
| 普通用户（C端） | 浏览商品、下单 | 浏览商品、提交订单（MVP 阶段简化） |

---

## 三、用户故事

### 3.1 演示场景

> **US-01** 作为演示者，我希望打开演示页面后能选择不同场景（订单流程/退款审批/上架审批），以便根据客户需求灵活展示。

> **US-02** 作为演示者，我希望每点击一次"下一步"，系统能推进一个流程节点并显示对应的业务变化，以便我能边操作边向客户讲解。

> **US-03** 作为演示者，我希望演示数据与正式数据隔离，且能一键重置，以便每次演示都从干净状态开始。

### 3.2 运营场景

> **US-04** 作为运营人员，我希望能在一个看板上看到所有待处理的订单和审批单，以便快速定位需要我操作的事项。

> **US-05** 作为运营人员，我希望点击订单卡牌上的操作按钮就能推进流程（如"确认发货"），而不需要跳转到多个页面。

> **US-06** 作为管理员，我希望审核运营提交的商品上架申请，同意后商品自动变为"已上架"状态。

> **US-07** 作为管理员，我希望能管理商品信息（新增、编辑、下架）和用户角色分配。

### 3.3 用户场景

> **US-08** 作为普通用户，我希望能浏览商品列表、查看商品详情，选择一个商品后填写收货地址并提交订单。

## 四、功能需求

### 4.1 工作流引擎（核心）

| 编号 | 功能 | 描述 |
|------|------|------|
| WF-01 | 流程模板管理 | 系统预置三个流程模板（订单全链路、退款审批、商品上架审批） |
| WF-02 | 流程实例创建 | 当业务事件发生时（下单/申请退款/提交上架），自动创建对应的流程实例 |
| WF-03 | 节点流转 | 支持手动推进流程到下一节点，校验流转规则（如已取消的订单不能发货） |
| WF-04 | 流转日志 | 记录每次流转的路径、操作人、时间、备注，支持审计追溯 |
| WF-05 | 多态关联 | 一套引擎同时关联订单、退款、商品等不同业务对象 |

### 4.2 演示模式

| 编号 | 功能 | 描述 |
|------|------|------|
| DM-01 | 场景选择 | 下拉切换三个预置场景（标准订单、退款审批、商品上架） |
| DM-02 | 分步推进 | 点击"下一步"推进流程，每次展示一个节点的变化 |
| DM-03 | 流程可视化 | 左侧垂直进度条显示所有节点及当前所处位置 |
| DM-04 | 节点详情 | 右侧展示当前节点的业务数据（订单金额、商品信息等） |
| DM-05 | 数据重置 | 一键重置演示数据到初始状态 |
| DM-06 | 数据隔离 | 演示数据独立存储，不影响正式数据 |

### 4.3 流程看板

| 编号 | 功能 | 描述 |
|------|------|------|
| DB-01 | 实例列表 | 卡牌式展示所有运行中的工作流实例 |
| DB-02 | Tab 切换 | 切换"订单流程"和"审批流程"两个视图 |
| DB-03 | 流程进度 | 每张卡牌显示已完成节点和当前节点 |
| DB-04 | 快捷操作 | 卡牌上提供"下一步"操作按钮 |
| DB-05 | 状态筛选 | 按流程状态（运行中/已完成/已取消）筛选 |

### 4.4 管理后台

| 编号 | 功能 | 描述 |
|------|------|------|
| AD-01 | 商品列表 | 分页展示所有商品，支持搜索 |
| AD-02 | 商品新增/编辑 | 表单录入商品名称、描述、价格、库存、分类、图片 |
| AD-03 | 商品删除 | 软删除商品（已关联订单的不可删） |
| AD-04 | 用户列表 | 展示所有用户及角色 |
| AD-05 | 角色分配 | 修改用户角色 |

### 4.5 用户端

| 编号 | 功能 | 描述 |
|------|------|------|
| US-01 | 商品浏览 | 商品列表页，支持分类筛选 |
| US-02 | 商品详情 | 展示商品大图、描述、价格、库存 |
| US-03 | 下单 | 填写收货地址，提交订单 |
| US-04 | 订单查看 | 查看自己的订单列表和状态 |

### 4.6 用户认证

| 编号 | 功能 | 描述 |
|------|------|------|
| AU-01 | 注册 | 邮箱 + 密码注册 |
| AU-02 | 登录 | 邮箱 + 密码登录 |
| AU-03 | 角色权限 | admin 可访问全部，operator 可访问看板和管理后台，customer 只能访问用户端 |

## 五、业务流程

### 5.1 订单全链路

```
用户下单 → 订单创建（待支付）
  → 用户模拟支付 → 已支付
  → 运营点击"发货" → 已发货
  → 用户点击"确认收货" → 已收货
  → 订单完成

异常分支：
  - 待支付状态：用户可取消 → 已取消
  - 已发货/已收货：用户可申请退款 → 退款流程启动
```

### 5.2 退款审批

```
用户申请退款 → 退款单创建（待审核）
  → 客服审核（通过/驳回）
  → 经理审批（通过/驳回）
  → 退款完成
```

### 5.3 商品上架审批

```
运营提交商品 → 状态变为"待审核"
  → 管理员审核（通过/驳回）
  → 通过后自动上架，用户端可见
```

## 六、验收标准

| 编号 | 验收条件 |
|------|----------|
| AC-01 | 打开演示模式，选择"标准订单流程"，能分5步完成从下单到收货的全链路 |
| AC-02 | 打开演示模式，选择"退款审批"，能分4步完成退款审批流程 |
| AC-03 | 流程看板能正确展示所有运行中的实例，Tab 切换正常 |
| AC-04 | 管理后台能正常新增、编辑、删除商品 |
| AC-05 | 用 customer 账号登录，能浏览商品并成功提交订单 |
| AC-06 | 用 operator 账号登录，能在看板上处理待办事项 |
| AC-07 | 用 admin 账号登录，能管理用户角色和审核商品上架 |
| AC-08 | 演示数据重置后恢复到初始状态 |
| AC-09 | 10 张数据库表通过 Prisma 迁移正常创建 |
| AC-10 | 种子数据启动后可正常登录三个预置账号 |

## 七、不在本期范围

- 真实支付渠道对接（微信/支付宝）—— 演示用模拟支付
- 物流 API 对接 —— 演示用模拟物流
- 可视化流程编辑器（拖拽配置）—— 二期
- 多商户/多仓库 —— 二期
- 短信/邮件通知 —— 二期
- 优惠券/积分/营销活动 —— 二期
- 数据报表/大屏 —— 二期

## 八、项目结构约定

```
ecom-flow/
├── prisma/                  # 数据库 schema + 迁移 + 种子
├── src/
│   ├── app/                 # Next.js App Router 页面
│   │   ├── (auth)/          # 登录/注册
│   │   ├── (shop)/          # 用户端（商品浏览/下单）
│   │   ├── dashboard/       # 流程看板
│   │   ├── demo/            # 演示模式
│   │   └── admin/           # 管理后台
│   ├── server/              # tRPC routers + 业务逻辑
│   │   ├── workflow/        # 工作流引擎
│   │   ├── product/         # 商品服务
│   │   ├── order/           # 订单服务
│   │   └── user/            # 用户服务
│   ├── lib/                 # 工具函数、Prisma client
│   └── components/          # 共享 UI 组件
├── docs/                    # 文档
└── docker-compose.yml       # PostgreSQL + Redis
```
